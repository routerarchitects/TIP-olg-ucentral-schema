# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repo is

A collection of JSON schemas defining the uCentral data model — the configuration
(`schema/`) and state (`state/`) objects exchanged with OpenLAN Gateway / uCentral
devices. This repo defines and validates the data model only; rendering configuration
JSON into device-specific output is done by separate flavour-specific renderers (the
renderer was dropped from `main` — see commit history).

## Source vs. generated artifacts

The YAML files under `schema/` and `state/` are the **only** hand-edited source.
Everything else listed below is generated from them and must be regenerated (never
hand-maintained in isolation) after a schema change:

- `ucentral.schema.json` — compact, `$ref`-based, descriptions stripped (validation)
- `ucentral.schema.pretty.json` — `$ref`-based, descriptions kept (docs)
- `ucentral.schema.full.json` — fully inlined (no `$ref`), descriptions kept
- `ucentral.state.pretty.json` — same, for the state model
- `validator/ucode/schemareader.uc` — autogenerated ucode validator/parser
- `validator/bash/schemareader.sh` — autogenerated jq-based bash validator, same role as
  the ucode one for environments without `ucode`

Regenerate with `./tools/generate.sh`, which runs `tools/merge-schema.py` for each JSON
target, `generators/ucode/generate-reader.uc` for the ucode validator, and
`generators/bash/generate-bash-reader.uc` for the bash validator. **Both validators must
be regenerated together** — they are independent generators reading the same
`ucentral.schema.pretty.json`, so a schema change that only reruns one of them silently
leaves the other out of sync.

## Schema authoring model

- Entry points: `schema/ucentral.yml` (config) and `state/state.yml` (state). Every
  other file is a sub-schema pulled in by reference.
- Cross-file references use absolute `$ref` URIs, e.g.
  `$ref: "https://ucentral.io/schema/v1/interface/ipv4/dhcp/"`. `tools/merge-schema.py`
  maps the URI path to a filename by replacing `/` with `.` and appending `yml`
  (so the ref above resolves to `schema/interface.ipv4.dhcp.yml`). File naming is
  therefore significant — keep the dotted-path convention.
- `format` keywords (e.g. `ipv4`, `uc-timeout`) drive validation in both generated
  validators — as `matchIpv4`, `matchUcTimeout`, etc. helpers in the ucode reader, and
  as branches of the `format_is_valid`/`format_desc` jq defs in the bash reader. When
  adding a field with a new `format`, both `generators/ucode/generate-reader.uc` and
  `generators/bash/generate-bash-reader.uc` need a matching case added.
- `schema.json` (root) is a hand-maintained `major.minor.patch` version of the schema,
  bumped independently of the generated files.

## Build / generation commands

```sh
./tools/generate.sh        # regenerate all JSON + both validators + docs HTML
./tools/merge-schema.py schema schema ucentral.yml ucentral.schema.json 1 1   # one JSON target
```

`tools/merge-schema.py` args: `<id> <dir> <entry.yml> <out.json> <tiny> <refs>` where
`tiny=1` strips descriptions/examples and `refs=1` emits `$ref`/`$defs` instead of
inlining.

External tools are required for a full `tools/generate.sh` run and are typically **not**
present in this dev environment: `ucode` (for `generators/ucode/generate-reader.uc` and
`generators/bash/generate-bash-reader.uc`), `jq` (to run the generated
`validator/bash/schemareader.sh`), `generate-schema-doc` (docs HTML). `python3` +
`pyyaml` (the only deps of `tools/merge-schema.py`) usually are present, so the JSON
targets can be regenerated even when `ucode` is missing.

### Gotcha: generate.sh truncates validator files on failure, and the two validators drift independently

`tools/generate.sh` uses `set -e` and shell redirects — `./generators/ucode/generate-reader.uc
> ./validator/ucode/schemareader.uc` and `./generators/bash/generate-bash-reader.uc >
./validator/bash/schemareader.sh`. Each redirect empties its target file **before** the
generator runs, so if `ucode` is absent (or a generator fails) that validator file is
left truncated.

`validator/ucode/schemareader.uc` and `validator/bash/schemareader.sh` are both
generated artifacts and must never be hand-edited — doing so drifts them from their
generators and is easy to get subtly wrong. They are also independent outputs of
independent generators, so regenerating one does **not** regenerate the other — a
schema change (e.g. a required-field change) must run through `tools/generate.sh`, not
just whichever single generator happens to be top of mind, or the other validator will
silently keep validating against the old schema. The fix when you hit truncation, or
when a schema change lands, is to install `ucode` and regenerate, not to patch a file by
hand:

- Install `ucode` from https://github.com/jow-/ucode (instructions:
  https://ucode.mein.io/#installation).
- Regenerate everything with `./tools/generate.sh`, or just one validator with
  `./generators/ucode/generate-reader.uc > ./validator/ucode/schemareader.uc` or
  `./generators/bash/generate-bash-reader.uc > ./validator/bash/schemareader.sh`.

If a run already truncated a file, `git checkout` it to restore, then regenerate once
`ucode` is available.

## Generated validators

This repo is schema-only — device runtime scripts (`command/*`, `system/*`) were
removed in the layout refactor and now live in the flavour-specific renderer repos.
What remains are two standalone, equivalent validators for checking a config/state JSON
document against the generated schema, each with its own generator (see above):

- `validator/ucode/schemareader.uc` — for environments with `ucode` available
- `validator/bash/schemareader.sh` — a jq-based equivalent for environments without `ucode`

## Testing

The README references `./run_tests.sh`, but that script is not present in the repo.
There is currently no runnable test harness checked in; validate changes by
regenerating the JSON and reviewing that the diff is localized to the intended section.
